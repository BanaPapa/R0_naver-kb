### 3-2. 월간 날짜축 파싱 규칙 (인제스트 핵심)

- A열에 연도가 **바뀌는 행에만** `'86.1`, `'13.4`, `2015.12` 형태로 표기되고, 이후 행은 월 숫자만
  (`2`,`3`,…)이다. → **연도 전파(carry-forward) 파싱** 필수.
- 표기가 시트마다 섞여 있다: `'86.1` / `2008.12` / `2013.4` / `2016. 1`(공백 포함) / `'08 1Q`(분기)
  / `'08.12·` (r4 라벨에 특수문자). 정규식 통합 파서가 필요.
- 일부 시트는 A열이 비고 B~C열에 라벨이 있는 경우(11·12 PIR의 분위 라벨)가 있어 시트별 컬럼
  오프셋 정의가 필요하다.

## 4. 상관관계 분석 결과

앱에 인제스트된 동일 원천 데이터(2026-06 기준 전 기간)로 계산. r = 피어슨 상관계수.

### A. 매매 ↔ 전세 동행성 (주간, 12주 변화율)

| 지역 | r | 해석 |
|---|---|---|
| 부산 / 대구 | **+0.92** | 지방 광역시는 매매·전세가 사실상 한 몸(실수요 일체형) |
| 세종 | +0.87 | 〃 |
| 전국 | +0.82 | - |
| 경기 | +0.69 | - |
| 강북14 | +0.61 / 서울 +0.58 / 강남11 +0.54 | **수도권 핵심지는 매매·전세 분리 국면 존재**(투자수요·규제 변수) |

→ 분석 시사점: "전세가 오르니 매매도 오른다"는 지방에선 성립하지만 서울 핵심지에선 절반만 맞다.
지역 탭 분석에서 이 동행성 차이를 리포트에 반영할 가치가 큼.

### B~C. 심리지수 ↔ 가격 증감 (주간, 리드-래그)

- 매수우위지수 vs 매매증감: 전국 최대 r=+0.84 @ **k=0~+2주(동행~미세 후행)**, 서울 +0.81 @ k=0.
- 전세수급지수 vs 전세증감: 전국 +0.79 @ k=0.
- → 두 지수는 **동행 확인 지표**다. 선행 예측력은 제한적이므로 "현재 국면 강도 측정"에 사용해야 함.

### D. KB 전망지수의 선행성 (월간) — **가장 유의미한 선행 신호**

| 지역 | 동월 r | **+1개월 r** | +2 | +3 | +6 |
|---|---|---|---|---|---|
| 전국 | +0.75 | **+0.82** | +0.78 | +0.73 | +0.59 |
| 서울 | +0.67 | **+0.77** | +0.64 | +0.51 | +0.34 |
| 경기 | +0.68 | **+0.76** | +0.73 | +0.69 | +0.55 |
| 부산 | +0.67 | **+0.73** | +0.70 | +0.66 | +0.55 |

→ 전망지수는 **다음 달 매매지수 증감과 최대 상관**. 1~3개월 단기 전망의 근거 지표로 유효.

### E. 전세가율 수준 vs 이후 12개월 매매 상승률 (월간)

전국 +0.07, 서울 +0.28, 경기 +0.11, 부산 +0.37 — **약한 상관**.
→ "전세가율이 높으면 갭투자로 상승" 가설은 수준(level)만으로는 근거 부족. 추세 전환과 결합 필요.

### F~G. 지역 간 전이 구조 (주간 매매 12주 변화율)

- **수도권 블록**: 서울↔경기 +0.82, 경기↔인천 +0.91 (서울→경기 **+2주**, 서울→인천 **+4주** 지연에서 최대)
- **지방 블록**: 부산↔대전 +0.82, 부산↔울산 +0.76, 대구↔울산 +0.75
- **블록 간 단절**: 서울↔부산 +0.29, 서울↔광주 +0.20 — 서울 상승이 지방으로 전이된다는 근거 약함
  (서울→부산 최대 r=+0.32 @ +10주로 미미)
- 세종은 서울과 +0.60으로 지방 중 유일하게 수도권과 중간 수준 연동.

→ 분석 시사점: 비교 분석 시 **같은 블록 내 비교**(서울-경기-인천 / 부산-울산-대전)가 의미 있고,
블록 간 비교는 "동행하지 않는다"는 사실 자체가 정보다.

## 5. 현재 앱과의 갭 분석

### 5-1. 데이터 파이프라인 (구조적 문제)

```
[KB 데이터허브]──(수동 다운로드)──▶ 엑셀 2개
                                     │  ❌ 변환 스크립트가 저장소에 없음 (재현 불가)
                                     ▼
                            public/data/kb-*.json (5개, 커밋된 산출물)
                                     │  scripts/kb-publish-bundles.mjs (수동 실행)
                                     ▼
                     Supabase Storage kb-data 버킷 (.gz + versions.json)
                                     │  앱 로드 시 버전 비교 → 변경분만 다운로드 (✅ 정상)
                                     ▼
                                  웹앱 (IndexedDB 캐시)
```

- **앱이 KB 사이트를 스스로 체크하지 않는다.** 자동 업데이트는 "Supabase에 새 번들이 올라온 뒤"
  클라이언트 배포 단계만 구현돼 있고, **엑셀 다운로드→변환→업로드는 전부 수동**이며 그 중
  변환 단계는 도구조차 저장소에 없다.
- 주간(매주)·월간(매월) 갱신 주기를 감안하면 이 수동 구간이 지속 운영의 최대 병목.

### 5-2. 인제스트된 데이터의 확인된 결함

1. **유령 지역 키**: `kb-weekly.json`에 `경기도|강원특별자치도도`, `경기도|춘천시`, `경기도|강릉시`,
   `경기도|원주시` 가 존재(값 전부 null). 강원 지역이 경기도 하위로 한 번 더 잘못 생성된
   파싱 잔재. 정상 키(`강원특별자치도|춘천시` 등)와 공존한다.
2. **신설 구 계층 평탄화**: 부천·화성 신설 구가 `경기도|원미구`, `경기도|동탄구` 처럼
   시(부천시/화성시) 계층 없이 도 바로 아래에 붙어 있다.
3. **2026 행정개편 지역 미포함**: 인천 4구·전남광주통합특별시가 지역 트리에 없다(현재 값은

4. **파일 간 지역 명칭 불일치**: 월간은 `전국>전라북도`(구명칭)·`전국>경기도>의왕`(시 누락),

## 6. 결론 요약

1. 두 파일은 "시트=지표, 행=시점, 열=지역" 기본 구조를 공유하나, **헤더 단수(1~4행), 지역 수
   (7~198), 시작 시점(1986~2016), 날짜 표기(시리얼/연도전파/분기)가 시트마다 제각각**이다.
   → 시트별 스키마 정의 테이블 기반의 범용 인제스터가 필요하다.
2. 지역 체계는 **열 순서 기반 계층 + 동명 구 + 명칭 세대 차이 + 2026 행정개편**이 겹쳐 있어
   정규화 테이블(canonical key) 없이는 파일 간·기간 간 정합이 깨진다. 현재 앱 데이터에 이미
   그 결함(유령 키, 명칭 불일치)이 확인됐다.
3. 상관분석 결과, **전망지수의 +1개월 선행성(r≈0.8)**과 **수도권/지방의 매매-전세 동행성 격차**,
   **서울→경기 2주 전이**가 이 데이터에서 뽑을 수 있는 핵심 구조적 신호다.
4. 파이프라인은 클라이언트 배포 구간만 자동화돼 있고 **수집·변환 구간은 수동+도구 부재**다.
   매주 운영을 감안하면 인제스트 스크립트의 저장소 내 재현 가능화가 최우선 과제다.
