# 작업 세션 기록 — KB 시계열 분석 앱 대개편 (2026-07)

> 이 문서는 한 번의 긴 작업 세션에서 진행한 모든 작업을 **학습 참고용**으로 정리한 것이다.
> "무엇을, 왜, 어떻게" 순서로 쓰되, 처음 보는 개념은 그때그때 풀어서 설명한다.
> 코드 세부는 각 커밋과 소스 주석에 있으니, 여기서는 큰 그림과 개념 위주로 남긴다.

---

## 0. 이 앱이 뭐였고, 무엇을 바꿨나 (한 문단 요약)

이 앱은 **KB부동산 시계열 통계**(매주·매월 발표되는 방대한 엑셀)를 읽어 지역별로 비교·분석하는
웹앱이다. 원래는 엑셀을 사람이 수동으로 받아 변환하고, AI 분석은 로컬 PC에서만 돌아갔다.
이번 세션에서 ① 데이터 오류를 바로잡고, ② 엑셀 다운로드~반영을 **완전 자동화**하고,
③ 분석 결과를 **A4 보고서로 인쇄/PDF**할 수 있게 하고, ④ 배포된 웹에서 **각 사용자가 자기 AI
키·구독으로** 분석하게 만들고, ⑤ 저장 데이터를 **계정에 연동**했다.

## 1. 등장한 핵심 개념 사전 (몰랐던 용어 위주)

| 용어 | 쉬운 설명 |
|---|---|
| **브랜치 / 머지** | 코드의 평행세계. `main`은 실제 배포되는 본선, 작업용 가지(feature 브랜치)에서 안전하게 만든 뒤 본선에 합치는(머지) 것. |
| **커밋 / 푸시** | 커밋=변경 저장점 찍기(로컬), 푸시=그걸 GitHub(원격)로 올리기. |
| **Vercel** | 코드를 GitHub에 올리면 자동으로 빌드해서 웹사이트로 배포해주는 서비스. `main`에 푸시하면 자동 재배포. |
| **정적 빌드 vs 서버 함수** | 정적=미리 만들어 둔 파일을 그대로 보여줌(빠름, 하지만 바꾸려면 재빌드). 서버 함수(Serverless Function)=요청이 올 때마다 서버에서 코드 실행(로그인 검증·외부 API 호출 등 동적 처리). |
| **Serverless(서버리스)** | "서버 없음"이 아니라 "내가 서버를 관리하지 않음". 요청 때만 잠깐 실행되고 사라지는 함수. **인스턴스 간 메모리를 공유하지 않는다**(중요 — 아래 OAuth에서 다시 나옴). |
| **Supabase** | 오픈소스 백엔드 서비스. 이 앱에선 ①로그인(Auth) ②데이터베이스(Postgres) ③파일저장(Storage)을 씀. |
| **RLS (Row Level Security)** | DB에서 "각 사용자는 자기 행(row)만 읽고 쓸 수 있다"를 DB 차원에서 강제하는 규칙. 앱 코드가 실수해도 남의 데이터에 못 닿게 하는 안전망. |
| **버킷 (Storage Bucket)** | 파일을 담는 폴더 같은 것. public이면 누구나 URL로 읽고, private이면 권한 필요. |
| **번들(bundle) / gzip** | 여러 데이터를 한 덩어리로 묶고(번들) 압축(gzip)해 전송량을 줄인 것. |
| **인제스트(ingest)** | 외부 원본(엑셀)을 앱이 쓰는 형식(JSON)으로 읽어들여 정제하는 과정. |
| **멱등(idempotent)** | 같은 작업을 여러 번 해도 결과가 같음. 자동 갱신을 매일 돌려도 안전한 이유. |
| **OAuth / PKCE** | 비밀번호를 넘기지 않고 "이 앱이 내 계정 일부를 쓰도록 허가"하는 표준 절차. PKCE는 그 과정을 앱에서 안전하게 하는 보강 기법. |
| **BYOK (Bring Your Own Key)** | "네 키는 네가 가져와라". 서비스가 AI 비용을 대신 내주는 게 아니라, 각 사용자가 자기 AI 키로 자기 비용을 쓰는 구조. |
| **환경변수 (Env Var)** | 코드에 직접 안 쓰고 바깥에서 주입하는 설정값(비밀키 등). `VITE_` 접두사는 브라우저까지 노출되는 값(비밀 아님), 접두사 없는 건 서버에서만. |

## 2. 진행한 작업 순서 (시간순)

### 2-1. 브라우저 확장 문구 정리 (법적 리스크 완화)
- 웹앱·확장의 사용자 노출 문구에서 외부 업체명(네이버·호갱노노)과 "크롤링/IP 차단" 같은
  위험 표현을 제거하고 중립적 표현으로 교체.
- 웹스토어 설치 버튼이 홈으로 가던 것은 버그가 아니라 **확장이 아직 스토어 미게시**라서였음.
  설치 URL을 환경변수(`VITE_EXTENSION_STORE_URL`)로 분리해 게시 후 코드 수정 없이 교체 가능하게.

### 2-2. A4 정형 보고서 + 인쇄/PDF
- 문제: AI 분석 결과가 모델마다 형식이 제각각이고, 출력(인쇄/PDF) 기능이 없었음.
- **근본 원인 발견**: 분석 실행기가 존재하지 않는 경로에서 프롬프트를 읽어, 형식 규칙 없이
  동작하고 있었음. 실제 프롬프트 파일을 읽도록 고침 → 모델 무관하게 같은 구조로 응답.
- `PrintReportView` 신규: 작성일·비교지역·결론·차트·근거·향후전망·용어설명을 갖춘 **A4 한 장**
  보고서. 날짜·차트·표는 데이터에서 직접 그려서, 모델이 형식을 어겨도 뼈대가 안 무너지게 설계.
- 브라우저 `window.print()`로 인쇄 → 대화상자에서 "PDF로 저장" 선택 가능.

### 2-3. 엑셀 2개 파일 완전 분석 (데이터 구조 파악)
- 주간(9시트)·월간(59시트) 엑셀 전체를 스크립트로 프로파일링.
- 발견: 시트마다 헤더 단수(1~4행), 지역 수(7~198), 시작 시점(1986~2016), 날짜 표기법이
  제각각. 2026 행정구역 개편(인천 구 개편, 전남광주통합) 컬럼이 빈 자리표시자로 미리 들어가 있음.
- **상관관계 분석**(전 기간 실측): 전망지수가 다음 달 매매증감의 최강 선행지표(r≈0.82),
  매매-전세 동행성이 지방(0.92) vs 서울핵심(0.54)로 갈림, 지역 전이는 블록 단위(수도권/지방).
- 산출: `docs/KB_TIMESERIES_DATA_REPORT.md`(구조 보고서), `docs/PROMPT_kb_analysis_upgrade.md`(작업 지시서).

### 2-4. 재현 가능한 인제스트 파이프라인 (P0)
- 문제: 엑셀→JSON 변환 도구가 **저장소에 없어서** 다음 주 갱신을 재현할 수 없었음.
- `scripts/kb-ingest.mjs`(엑셀→JSON 결정적 재생성 + 기존과 diff 비교) +
  `scripts/kb-regions.mjs`(지역명 정규화: 동명 구 처리, 신구 명칭 통일, 오타 흡수).
- **데이터 결함 수정**: `경기도|춘천시` 같은 유령 키 4건 발견 → 원인은 KB 원본 엑셀의 오타
  (`강원특별자치도도`). 정규화로 흡수해 춘천 전세 데이터가 제대로 연결되게 함.
- 검증: 기존 데이터와 값 비교 수십만 건에서 불일치 0건(파서가 기존을 정확히 재현).

### 2-5. 데이터셋 확장 → 그리고 되돌림 (교훈)
- 한때 주택유형(종합/단독/연립)·중위가격·선도50지수를 추가했으나, 사용자가 "무거워지고 의미
  없다"며 제거 요청 → **전부 되돌려 아파트 전용으로 복원**. 화면·데이터가 main과 동일해짐.
- 교훈: 데이터를 늘리는 것보다 **기존 데이터의 오류 수정과 관계(상관) 발견**이 더 가치 있었음.
- 상관분석 지식은 AI 분석 프롬프트에 도메인 지식으로 주입(전망지수 선행성 등)해서 남김.

### 2-6. 완전 자동 갱신 (수동 → 무인)
- 조사: KB 통계 API(`api.kbland.kr/land-extra/statistics/reference`, `getfiledown`)가
  **로그인 없이** 열려 있음을 브라우저로 확인 → 스크립트로 직접 다운로드 가능.
- `scripts/kb-update.mjs`: 확인→다운로드→인제스트→발행을 한 번에(멱등).
- **처음엔 이 PC의 Windows 작업 스케줄러**에 등록했으나, "여러 PC를 오간다"는 사용자 상황에
  맞춰 **GitHub Actions로 이전**(`.github/workflows/kb-data-update.yml`). 이제 **어떤 PC도
  켜져 있을 필요 없이** 깃허브 클라우드가 매일 자동 실행 → 새 파일이면 커밋 → Vercel 자동 재배포.

### 2-7. KB 데이터를 Supabase Storage로 배포
- 발행 스크립트(`kb-publish-bundles.mjs`)가 gzip 번들을 버킷에 올리고 버전 매니페스트 갱신.
- 버킷을 **공개(public)로 전환**(공개 통계라 비밀 아님) → 익명 사용자도 URL로 읽기 가능.
- Vercel 환경변수 `VITE_KB_DATA_SOURCE=supabase`를 켜면 앱이 버킷에서 변경분만 받아옴
  (재배포 없이 데이터만 갱신). 안 켜면 정적 모드(빌드 포함 JSON) 그대로.

### 2-8. 사용자 저장 데이터 계정 연동
- 문제: 저장한 분석·차트 슬롯이 **브라우저 localStorage에만** 있어 다른 기기에서 안 보임.
- `kb_user_state` 테이블(계정당 1행, RLS 본인 전용) + `user-data-sync`:
  로그인 시 클라우드↔로컬 **병합**(분석은 id 합집합, 슬롯은 최신 우선), 이후 변경은 디바운스 업로드.
  기존 브라우저 데이터는 첫 로그인 때 자동으로 계정에 이관. 비로그인은 기존 로컬 동작 유지.

### 2-9. 배포 환경 AI 분석 — BYOK + 저장 선택 + 구독 인증
이번 세션의 마지막이자 가장 복잡한 부분. 세 겹으로 정리한다.

**(a) 문제**: AI 분석은 로컬 개발용 "vite 브리지"에만 있었음. 배포된 웹에는 그 브리지가 없어
분석 기능이 깨져 있었음. 사용자 의도는 "**로그인한 각자가 자기 키·구독으로** 분석"(공유 아님).

**(b) BYOK 서버리스**: `api/kb-analysis.ts`(서버 함수)가 사용자를 검증하고, 그 사용자의
자격증명으로 AI를 호출. 키는 응답에 절대 안 나감. OpenAI호환·Anthropic·Gemini 어댑터를 인라인.

**(c) 저장 선택(사용자 요청)**: "키를 서버에 저장하기 싫은 사람"을 위해 두 모드:
- **이번만 사용**(기본): `sessionStorage`(브라우저 탭)에만 보관, 서버 무저장. 분석할 때마다
  자격증명을 요청에 실어 보냄. 탭 닫으면 사라짐.
- **내 계정에 저장**: Supabase `kb_user_providers`(RLS 본인 전용)에 저장, 기기 간 공유.

**(d) 구독 인증(사용자 요청, P1_Reviewer 참고)**: 여기서 중요한 개념 정정이 있었다.
- 처음엔 "구독 OAuth는 콜백이 localhost라 배포 불가"라고 판단 → **틀렸음**.
- 사용자가 이미 배포 성공한 `P1_Reviewer` 레포를 참고로 제시. 분석해보니 **콜백이 필요 없는
  두 방식**을 쓰고 있었다:
  - **device-code**(OpenAI): 앱이 1회용 코드를 보여주고, 사용자가 별도 승인 페이지에서
    그 코드를 승인하면, 앱이 폴링(주기적 확인)으로 토큰을 받음. **콜백 URL 자체가 불필요**.
  - **oauth-code**(xAI): 동의 화면이 코드를 화면에 표시 → 사용자가 복사해서 앱에 붙여넣기.
- 서버리스는 인스턴스 간 메모리를 공유 안 하므로, PKCE 임시값을 **HMAC 서명 쿠키**로 왕복시킴
  (P1의 핵심 트릭). `api/kb-oauth.ts`가 device start/poll·oauth start/exchange를 담당.
- 교환된 토큰도 저장 선택(이번만/계정)을 따름. OpenAI 구독은 ChatGPT Codex 백엔드로 실행.

## 3. 지금의 전체 아키텍처 (한눈에)

```
┌─ 데이터 파이프라인 (완전 자동, PC 불필요) ─────────────────┐
│  GitHub Actions(매일 10:30) → KB API 확인 → 새 파일 다운로드  │
│    → kb-ingest(JSON 변환·검증) → ① public/data 커밋           │
│                                    → Vercel 자동 재배포(정적) │
│                                 → ② Supabase 버킷 발행(선택)  │
│                                    → 앱이 변경분만 다운로드    │
└──────────────────────────────────────────────────────────┘

┌─ 사용자별 데이터 (Supabase, RLS 본인 전용) ────────────────┐
│  로그인 계정                                                │
│   ├ kb_user_state      : 저장한 분석·차트 슬롯(기기 간 공유) │
│   └ kb_user_providers  : (계정 저장 모드) AI 키·구독 토큰    │
│  브라우저(sessionStorage): (이번만 모드) AI 키·구독 토큰      │
└──────────────────────────────────────────────────────────┘

┌─ AI 분석 실행 ────────────────────────────────────────────┐
│  개발(로컬): vite 브리지 — 파일 저장 + loopback OAuth         │
│  배포: api/kb-analysis(BYOK) — 각자 키/구독으로 실행         │
│        api/kb-oauth — device-code·oauth-code 구독 인증       │
└──────────────────────────────────────────────────────────┘
```

## 4. 사용자가 직접 해야 하는 설정 (코드로 안 되는 것)

이런 건 보안·권한 때문에 대시보드에서 사람이 직접 해야 한다. 정리:

1. **Supabase SQL Editor에서 마이그레이션 3개 실행** (`supabase/migrations/*.sql`):
   - `kb_user_state` (저장 데이터), `kb_user_providers` (AI 키/구독), 버킷 읽기 정책.
2. **Vercel 환경변수**:
   - `VITE_KB_DATA_SOURCE=supabase` (버킷에서 데이터 읽기; 설정 후 Redeploy 필요)
   - `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` (로그인·DB; 이미 있을 것)
   - `CRAWL_TOKEN_SECRET`, (선택) `OAUTH_SECRET` (서버 서명용 비밀)
3. **GitHub Secrets** (Settings → Secrets → Actions): `SUPABASE_URL`,
   `SUPABASE_SERVICE_ROLE_KEY` (Actions가 버킷 발행할 때 필요; 없으면 커밋 경로만 동작).
4. **`.env.kb-publish`** (로컬에서 수동 발행할 때만; git에 안 올라감).

> 팁: `VITE_` 접두사 = 브라우저에 노출돼도 되는 값(anon 키는 원래 공개용). service_role 키는
> **절대** 브라우저·git에 노출 금지(관리자 키). 그래서 이건 서버(Actions/함수)에만 둔다.

## 5. 앞으로 남은 것 / 주의점

- **브라우저 확장 웹스토어 게시**: 매물 검색은 확장이 필요한데 아직 미게시. 게시(개발자 계정
  $5 + 심사) 후 `VITE_EXTENSION_STORE_URL`만 실제 URL로 바꾸면 됨.
- **service_role 키 노출 주의**: 세션 중 스크린샷에 한 번 찍혔음. 찜찜하면 Supabase에서
  키 재발급(rotate) 후 관련 위치(GitHub Secret, `.env.kb-publish`) 갱신.
- **구독 인증은 실험적**: OpenAI/xAI의 비공식 CLI OAuth 흐름을 이용 → 제공사 정책 변경 시
  깨질 수 있음. API 키 방식이 가장 안정적.
- **PC 의존성 없음 확인**: 자동 갱신은 GitHub Actions로 이전했으므로 이 노트북을 꺼도/폴더를
  지워도 데이터 갱신은 계속됨.
